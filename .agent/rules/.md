# Rule: Lovable & Mistral Full-Stack Standards
description: Core architectural and implementation guidelines for the Ambient EHR Agent hackathon project using Lovable, FastAPI, Mistral, Voxtral, ElevenLabs, FHIR, and Epic on FHIR.

1. **Frontend (Lovable):**
   - Build a minimalist, reactive interface focused on microphone/audio upload, live transcript visibility, extracted entity review, and final prior-auth/FHIR export.
   - Route live audio through browser-native capture flows optimized for WebSocket streaming, while preserving a separate fallback path for batch audio upload.
   - Keep the UI presentation-only: render transcript, entities, and final outputs without embedding extraction, validation, or orchestration logic in the frontend.

2. **Backend (FastAPI):**
   - Structure all request handlers asynchronously and keep a strict separation between the realtime WebSocket ingestion route (`/ws/realtime-transcribe`) and the REST processing route (`POST /process-clinical-data`).
   - Centralize workflow control in an explicit orchestrator/state machine that manages transcript updates, extraction, clarification loops, serialization, persistence, and export.
   - Ensure manual clarification text, uploaded audio fallback, and realtime transcript chunks all converge through the orchestrator rather than bypassing the application flow.

3. **AI Integration (Mistral + Voice):**
   - Use **Voxtral Realtime** as the primary live ASR path and **Voxtral Mini Transcribe V2** as the resilience fallback for uploaded or reprocessed audio.
   - Use **Mistral Large 3** only for reasoning-heavy tasks such as clinical entity extraction, missing-field detection, and prior-authorization logic; enforce explicit structured outputs or tool-calling patterns.
   - Restrict **Ministral 3 8B** to deterministic serialization only, converting validated entities into strict prior-auth JSON and FHIR-aligned JSON.
   - Use **ElevenLabs** exclusively for concise clarification prompts when required entities are missing; keep the voice loop short, targeted, and tied to validation outcomes.

4. **Validation & Data Integrity:**
   - Validate all extracted and serialized outputs with **Pydantic schemas** before any persistence, export, or sandbox submission.
   - Perform required-entity checks before serialization; do not generate final payloads until mandatory clinical and administrative fields are present.
   - Treat FHIR-aligned JSON generation as a controlled formatting stage after validation, not as an unconstrained free-text model response.

5. **Persistence & Integration:**
   - Persist session state, logs, and intermediate artifacts in **SQLite** so each run remains traceable and recoverable during demos.
   - Submit only validated FHIR-aligned payloads to the **Epic on FHIR Sandbox** to simulate downstream interoperability and EHR injection.
   - Keep external integrations thin and replaceable so the live demo remains stable even if the sandbox or network path fails.

6. **Build & Training Tooling:**
   - Use **Synthea** to generate synthetic FHIR-compatible clinical data for demos, schema testing, and optional serializer fine-tuning.
   - Keep the fine-tuning pipeline separate from runtime inference; treat any fine-tuned checkpoint for Ministral as optional optimization, not a hard dependency for the live flow.
   - Use **Mistral Vibe CLI** only as a development accelerator for scaffolding backend and UI integration, never as a runtime system component.
